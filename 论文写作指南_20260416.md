2. **核心技术方法**（2-3句）：系统采用"混合推荐架构"——语义检索基于BGE-M3向量检索与ChromaDB实现语义匹配，内容匹配基于FMA元数据（流派/情绪/能量/标签）实现内容相似度匹配。通过LLM Agent实现意图识别→槽位填充→工具调用的自动化编排，并利用RAG技术增强回复的可解释性，**每首推荐歌曲均附带LLM生成的个性化推荐理由**。

## 第1章 课题背景及意义

### 1.1 课题背景

**写作要点**：

1. **行业背景**：互联网音乐平台（Spotify、网易云音乐）的发展，推荐系统从单轮转向多轮对话推荐（对话式推荐，Conversational Recommender System）
2. **技术背景**：LLM的爆发（GPT-4/Qwen），大模型在理解用户意图和生成自然语言方面的优势
3. **实际需求痛点**：
   - 传统推荐系统"猜你喜欢"缺乏可解释性和交互性
   - 用户难以精确表达偏好，只能被动接受推荐结果
   - 协同过滤的"冷启动"和" popularity bias"问题
4. **引入本课题**：提出基于LLM的多轮对话音乐推荐助手，通过自然语言交互理解用户意图，结合混合推荐架构实现精准可解释的推荐

**建议引用**：
- Ricci et al. (2021) "Conversational Recommender Systems" 对话式推荐系统的定义与挑战
- 网易云音乐2023年报告：多轮对话使用户留存率提升23%

### 1.2 国内外研究现状

**写作结构**：

1. **对话式推荐系统研究**（3-4段）
   - Sun et al. (2018) "Next Item Recommendation with Conversational bandits" 对话式 bandits
   - Christakopoulos et al. (2015) "A survey of conversational recommender systems"
   - 国内：清华"灵犀"、北大对话推荐系统研究

2. **音乐推荐算法研究**（3-4段）
   - **基于内容的推荐（Content-Based）**：Baltrunas et al. (2011) "Item-based top-N recommendation" —— 基于内容特征的相似度计算；本系统内容匹配采用FMA元数据（流派/情绪/能量/标签）构建歌曲内容特征向量，解决协同过滤的冷启动问题
   - 协同过滤：Koren et al. (2009) "Matrix Factorization Techniques for Recommender Systems" —— 传统协同过滤的矩阵分解方法
   - 混合推荐：Burke (2002) "Hybrid Recommender Systems" —— 混合推荐系统综述，融合多种策略提升推荐质量
   - 基于音频特征的推荐：Spotify使用音频特征（MFCCs, Chroma）进行音乐推荐；Van Mn et al. (2013) 利用深度学习提取音频特征进行音乐推荐

3. **LLM+推荐系统研究**（3-4段）
   - ChatGPT-Recommender (2023)：LLM作为推荐引擎
   - RAG（Retrieval-Augmented Generation）在推荐系统中的应用
   - 大模型意图识别与槽位填充研究

4. **总结评述**：指出当前研究的不足（LLM推理成本高、多轮上下文建模复杂、双通道融合策略不完善、传统CF冷启动问题），引出本课题的改进方向

### 1.3 主要研究内容

**建议列出3-5个核心研究内容**：

1. **多轮对话管理机制**：设计基于LLM的意图识别与槽位填充算法，实现上下文状态维护与指代消解
2. **混合推荐架构**：融合语义向量检索（BGE-M3+ChromaDB）与基于FMA元数据的内容相似度匹配，提出加权融合策略，有效解决冷启动问题
3. **LLM生成推荐理由**：利用RAG技术构建音乐知识上下文，每首推荐歌曲附带15-40字的个性化推荐理由，帮助用户理解推荐依据
4. **用户反馈闭环**：实现Like/Dislike/换一批等交互行为对推荐结果的动态调整，构建偏好学习机制
5. **会话数据持久化**：基于SQLite实现跨会话历史存储与多用户数据隔离，重启后推荐历史不丢失

#### 2.1.1 FMA 数据集介绍

**FMA（Free Music Archive）** 是一个开放的音乐数据集，由 Deffner 等人于2017年提出，旨在为音乐推荐研究提供标准化的数据集。FMA 提供多个规模的子集，其中 **FMA Small** 包含 8,000 首完整音轨及其元数据。

**本系统使用的数据规模：**

| 指标 | 数值 |

| 总歌曲数 | 106,573 首 |

| 扩展字段 | rich_text, genres_all, artist_tags, track_tags |
| 时间范围 | 开放音乐档案（2000年代至今） |

**写作要点**：
1. FMA 的优势：CC 协议开放、丰富的流派标注、标准化的元数据结构
2. 本系统的扩展：融合了 Last.fm 的用户标签数据作为补充
3. 选取 FMA 而非 Million Playlist Dataset 的原因：流派标注更细粒度，适合内容推荐

### 2.4 数据向量化与索引构建

#### 2.4.1 BGE-M3 模型介绍

**BGE-M3**（BAAI General Embedding Model）是北京人工智能研究院开源的多语言向量模型：
- **支持语言**：中、英、法、德、日、韩等 100+ 语言
- **维度**：1024 维向量
- **特点**：无需额外训练即可处理中英文混合查询

**为何选择 BGE-M3：**
1. 多语言支持：用户中文输入可直接编码，无需翻译
2. 开源免费：无需调用商业 API
3. 高效：单次编码 < 50ms，适合在线检索

#### 2.4.2 歌曲文本表示

每首歌曲的检索文本由以下字段拼接而成：

```
检索文本 = title + " " + artist + " " + genre + " " + tags + " " + mood_tags + " " + scene_tags
示例："Midnight Drive Apollo Rising Electronic energetic futuristic night-drive driving"
```

#### 2.4.3 ChromaDB 索引构建

**ChromaDB** 是一个轻量级的向量数据库，支持：
- 高效的相似度检索（余弦相似度）
- 元数据过滤
- 增量索引更新

**索引参数：**

| 参数 | 值 | 说明 |
|------|---|------|
| 向量维度 | 1024 | BGE-M3 输出维度 |
| 距离度量 | 余弦相似度 | cosine similarity |
| 索引类型 | HNSW | Hierarchical NSW |
| 歌曲规模 | 106,573 | 全部入库 |

##### 5.2.2 混合推荐模块（双通道互补策略）

**核心设计**：语义检索（Semantic Search）+ 内容匹配（Content-Based Matching）双路召回，互补融合。

**语义检索通道**：
- 基于 BGE-M3 向量模型 + ChromaDB 实现
- 用户查询文本（中/英文）直接编码为 1024 维向量，无需翻译
- 通过余弦相似度在 106,573 首歌曲的向量索引中检索最相关结果
- 适用场景：语义模糊查询（如"适合学习的音乐"、"让我放松的歌"）

**内容匹配通道**：
- 基于 FMA 元数据（流派/情绪/能量/标签）构建歌曲内容特征
- 计算歌曲间余弦相似度，提供多样性补充推荐
- 有效缓解语义检索"同质化"问题
- 适用场景：精确流派/情绪需求、补充语义检索的长尾结果

**融合策略**：加权融合（默认 0.6:0.4），综合语义相关性 + 内容多样性

**技术说明（论文中如何描述）**：

> 本系统的内容匹配采用基于内容特征的推荐策略（Content-Based Filtering）。具体而言，系统以FMA数据集的元数据（流派、情绪、能量、标签等）作为歌曲的内容特征表示，通过计算歌曲间的余弦相似度构建相似度矩阵。该方法不依赖用户行为数据即可冷启动，有效解决了传统协同过滤在新用户/新歌曲上的冷启动问题。当一路推荐无法返回足够结果时，另一路补充召回，形成互补策略，保证推荐结果的稳定性。

#### 5.2.3 用户画像模块（AI生成深度解读）

**设计目标**：基于用户历史行为与对话记录，生成个性化音乐品味解读。

**核心功能**：
- **多维度统计**：流派偏好计数、能量偏好加权平均、情感标签分布
- **对话历史挖掘**：提取最近10轮对话意图，作为场景偏好参考
- **AI深度分析**：由大语言模型（Qwen）生成15-40字的个性化品味解读

**数据来源**：
- 用户喜欢/不喜欢歌曲的流派分布（liked_genre_counts）
- 歌曲的能量评分历史（energy_scores 加权平均）
- 最近对话场景偏好（从 SessionPersistence 读取）
- 最近对话记录（从 ChatHistory 读取最近10轮）

**Deep Analysis 生成流程**：

```
用户请求画像
      ↓
读取喜欢的歌曲列表（含流派元数据）
      ↓
从 ChromaDB 补充歌曲流派信息
      ↓
统计流派分布、能量偏好
      ↓
读取最近10轮对话（ChatHistory）
      ↓
读取当前场景/情感偏好（SessionPersistence）
      ↓
组装 Prompt → 调用 Qwen LLM
      ↓
生成个性化品味解读（缓存至 DB）
```

**缓存机制**：Deep Analysis 结果存入 `user_preferences.ai_portrait_deep_analysis` 字段，仅在字段为空时调用 LLM，避免重复开销。

**技术说明（论文中如何描述）**：

> 本系统的用户画像模块不仅统计用户的表面偏好数据（流派计数、能量评分），还利用大语言模型对用户的音乐品味进行深度解读。系统综合用户的历史喜欢歌曲列表、最近对话场景偏好以及最近对话记录，组装为结构化 Prompt，调用 Qwen 大模型生成一段贴合用户实际音乐消费场景的个性化解读。该解读作为用户画像的"AI总结"部分，帮助用户直观理解自己的音乐偏好特征。

#### 5.2.4 会话持久化模块

**设计目标**：解决传统对话系统"重启后对话历史丢失"问题，实现跨会话状态恢复。

**持久化数据**：
- `session_persistence` 表：存储会话级状态（当前场景/流派/情绪/能量偏好、已喜欢/不喜欢歌曲ID列表、推荐结果排除ID列表）
- `chat_history` 表：存储每轮对话详情（用户输入、系统回复、意图、推荐结果）
- `user_preferences` 表：存储用户级偏好（流派计数、能量偏好、AI画像解读）

**时间戳处理**：
- 数据库存储 Unix 时间戳（float），前端 JS 通过乘以 1000 转换为 Date 对象
- 避免 `datetime` 对象序列化时的时区歧义问题
- 会话创建时间以 `session_persistence.created_at` 为权威来源

**会话删除逻辑**：删除会话时同时清理 `chat_history` 和 `session_persistence` 两条记录，避免空会话残留。

**技术说明（论文中如何描述）**：

> 本系统的会话持久化机制基于 SQLite 实现。不同于传统对话系统仅存储文本对话历史，本系统还额外持久化了会话级的推荐状态（已推荐歌曲ID、已排除歌曲ID）和用户偏好快照（当前场景/流派/情绪/能量）。系统重启后，用户可从历史会话列表中加载任意会话，系统根据 `session_persistence` 中的偏好快照恢复推荐状态，实现无缝继续。Unix 时间戳的使用确保了前后端时间表示的一致性，避免了时区解析歧义。

#### 5.2.5 用户偏好与反馈闭环模块

**设计目标**：让用户反馈真正影响推荐结果，形成闭环

**模型设计**（`UserPreference`）：

```python
class UserPreference(Base):
    user_id = Column(Integer, primary_key=True)
    
    # 喜欢的流派及计数：{"Rock": 5, "Jazz": 3}
    liked_genre_counts = Column(JSON, default=dict)
    
    # 不喜欢的流派及计数：{"Metal": 2}
    disliked_genre_counts = Column(JSON, default=dict)
    
    # 能量偏好：使用加权平均而非覆盖
    energy_scores = Column(JSON, default=dict)  # {"scores": [3,2,3], "count": 3}
    
    # AI画像深度解读（由Qwen生成，缓存于DB）
    ai_portrait_deep_analysis = Column(String(2000))
```

**关键设计决策**：

| 决策 | 方案 | 理由 |
|------|------|------|
| 存储方式 | Counter 计数 | 避免列表重复问题 |
| 能量处理 | 加权平均 | 直接覆盖不合理 |
| 融合方式 | 加法融合 | 乘法可能溢出/归零 |
| Dislike阈值 | 3次 | 避免1次误判 |

**反馈处理流程**：

```
用户点击 Like
      ↓
记录行为到 user_behaviors 表
      ↓
更新 UserPreference.liked_genre_counts[genre]++
      ↓
更新 UserPreference.energy_scores（加权平均）
      ↓
更新"我喜欢的音乐"播放列表
      ↓
下次推荐时应用加法融合

用户点击 Dislike
      ↓
记录行为到 user_behaviors 表
      ↓
更新 UserPreference.disliked_genre_counts[genre]++
      ↓
下次推荐时应用阈值判断（≥3次才降权）
```

**推荐融合公式**：

```
最终分数 = 语义相似度 + 偏好调整值

偏好调整值 = 
    +min(0.15, liked_count * 0.03)    # 喜欢加分
    -min(0.20, disliked_count * 0.05)   # 不喜欢减分（需≥3次）
```

**技术说明（论文中如何描述）**：

> 本系统的用户偏好模块采用计数机制记录用户反馈。不同于传统推荐系统简单记录"喜欢/不喜欢"，本系统将反馈聚合为流派维度的偏好强度（liked_genre_counts、disliked_genre_counts）。在推荐排序阶段，使用加法融合策略：根据用户对某流派的累计反馈次数，在基础语义相似度上叠加偏好调整值（+0.03~+0.15 或 -0.15~-0.20）。该方法既保证了用户偏好的累积效应，又通过阈值机制（3次）避免了单次误触发的偏见。

### 6.3 关键工程问题与解决方案

#### 问题1：LLM模式反馈处理分支绕过DB写入

**问题描述**：当 `LLM_MODE="qwen"` 且 `generated_by_llm=True` 时，反馈处理器的 LLM 分支在调用 `state.add_feedback()` 后直接返回响应，跳过了所有数据库写入操作（`record_behavior` 表记录、`UserPreference.liked_genre_counts` 更新、"我喜欢的音乐"播放列表添加）。这导致用户偏好数据全部丢失。

**根因**：LLM 分支的早期返回逻辑（return statement）跨越了包含所有 DB 写入操作的 fallback 分支。Fallback 分支原本设计为"LLM 调用失败时的降级路径"，而非"LLM 成功时也必须执行的写入路径"。

**解决方案**：在 LLM 分支的成功路径中显式添加所有必要的 DB 写入操作，包括：
- `record_behavior()` 行为记录写入
- `UserPreference.record_like()` / `record_dislike()` 偏好更新
- "我喜欢的音乐"播放列表歌曲添加

**技术说明（论文中如何描述）**：

> 在实现 LLM Agent 反馈处理时，发现了一个关键的结构性缺陷：反馈处理函数中存在两个平行分支——LLM 分支（处理 LLM 生成回复的场景）和 Fallback 分支（处理 LLM 调用失败或非 LLM 场景）。原实现中，LLM 分支在完成内存状态更新后直接返回响应，而将所有 DB 写入操作放在了 Fallback 分支中。这导致在 LLM 模式下，所有用户偏好数据均无法持久化。本系统通过将 DB 写入操作显式添加到 LLM 分支中，确保了无论采用何种生成模式，用户反馈均能正确记录到数据库，形成完整的偏好学习闭环。

#### 问题2：会话时间戳6小时时区偏移

**问题描述**：前端收到的会话创建时间比实际时间晚6小时，导致历史会话列表中的时间显示错误。

**根因**：后端使用 `datetime` 对象（含时区信息 UTC）进行序列化，前端 JavaScript 的 `new Date()` 将 UTC 时间误当作本地时区进行解析。服务器时区为 UTC+6（CST），导致时间偏移6小时。

**解决方案**：后端统一使用 Unix 时间戳（float）作为时间表示，存储和传输均使用 `time.time()` 的返回值。前端通过 `new Date(timestamp * 1000)` 进行解析，彻底消除时区歧义。

#### 问题3：用户画像 Deep Analysis 重复调用 LLM

**问题描述**：每次请求用户画像时都会调用 Qwen LLM 生成深度解读，造成不必要的 API 调用开销和成本。

**根因**：`GET /portrait` 接口未对深度解读结果进行缓存，每次请求均触发新的 LLM 调用。

**解决方案**：在 `UserPreference` 模型中新增 `ai_portrait_deep_analysis` 字段（String, 2000字符），LLM 生成结果后立即写入数据库。下次请求时，若该字段非空则直接返回，不重复调用 LLM。数据库变更通过迁移脚本完成（`migrate_add_portrait_deep_analysis.py`）。

#### 问题4：空会话删除返回404

**问题描述**：当某个会话存在 `session_persistence` 记录但没有任何聊天历史时，调用删除接口返回404错误。

**根因**：`delete_session` 函数仅检查 `ChatHistory` 表中是否存在会话记录，未检查 `SessionPersistence` 表。

**解决方案**：删除会话时同时检查 `ChatHistory` 和 `SessionPersistence` 两张表，确保无论会话是否有聊天历史，均能正确删除。
