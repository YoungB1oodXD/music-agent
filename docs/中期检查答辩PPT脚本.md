**台词**：

## 第 5 页：系统架构

**标题**：系统是怎么工作的？

**架构图**（简化版）：
```
用户说话 → 理解意图 → 找歌曲 → 生成解释 → 返回给用户
           ↓
        记住偏好
```

**内容**：
- 左边：理解用户说的自然语言（BGE-M3）
- 右边：基于用户行为推荐（协同过滤）
- 中间：大模型调度，生成解释

**台词**：
> 系统分三个部分。左边是语义理解，用向量模型把用户说的话转成数学表示，然后去数据库里找相似的歌曲。右边是协同过滤，基于用户行为做推荐，类似于"喜欢这首歌的人还喜欢"。中间是大模型，负责理解用户意图、调度工具、生成推荐理由。

## 第 8 页：核心功能 - 多轮对话

**标题**：多轮对话状态管理

**内容**：
- 系统记住之前的对话
- 偏好可以累积和调整
- 用户说"换一批"，系统在当前偏好下刷新

**示例对话流程**：
1. 用户：推荐适合学习的歌
2. 系统：返回 5 首
3. 用户：更欢快一点，适合跑步
4. 系统：返回新的 5 首（场景切换）
5. 用户：不要人声
6. 系统：返回纯音乐（叠加偏好）

**台词**：
> 多轮对话是核心功能。系统会记住之前的对话内容。比如用户先说"推荐适合学习的歌"，系统返回结果后，用户可以继续说"更欢快一点，适合跑步"，系统会把场景从学习切换到跑步，而不是重新开始一轮对话。这就是多轮对话状态管理。

**标题**：用户反馈闭环

**内容**：

- 不喜欢：排除该歌曲

## 备用问答

**Q: 为什么协同过滤没用上？**
> 协同过滤模块已经实现了，代码也有。问题是协同过滤模型是在 Last.fm 数据上训练的，语义检索用的是 FMA 数据，两者的歌曲 ID 不匹配。这是数据层面的工程问题，不是算法问题。我计划在后续论文工作中重新对齐数据。

**Q: 为什么音频试听只完成了60%？**
> FMA_small 数据集的音频文件 ID 和推荐结果的 ID 存在对齐问题。我构建了一个映射文件，覆盖了大约8000首歌，但不是全部。答辩时我会把这个作为系统局限性说明。

**Q: 有没有做过推荐效果评估？**
> 目前还没有做离线评估，比如准确率、召回率这些指标。这是一个不足，我计划在答辩前补充，或者作为后续优化方向在论文中说明。

**Q: 大模型是实时的还是本地的？**
> 大模型用的是阿里云的 Qwen API，需要联网。但我同时实现了一个 Mock 模式，用预定义的规则来模拟大模型行为，确保在没有 API Key 或者断网的环境下，核心逻辑仍然可以验证和演示。

**Q: 多轮对话能记住多少轮？**
> 目前设计的是记住最近5轮对话。状态会话存在内存里，重启服务会清空。持久化存储可以后续加。

**Q: 这个系统和现有的音乐推荐有什么区别？**
> 最大的区别是对话主导。用户不需要点按钮筛选，直接说话就行。系统会理解意图、记住偏好、持续调整。传统推荐更多是被动的列表推送。
