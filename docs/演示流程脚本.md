# 演示流程脚本

> 适用于毕业设计中期检查、开题答辩、功能演示等场景  
> 预计演示时间：每个流程 8-12 分钟

---

## 流程一：学习场景 - 从安静到爵士

**演示重点**：场景识别 → 情绪调整 → 曲风细化 → 反馈机制 → 换一批去重

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**推荐一些适合深夜学习的音乐** | Loading 显示 5 个阶段（每阶段 3s），停在"精选歌曲"等待<br>返回 5 首安静/专注风格的音乐 | 展示 Loading 动效<br>展示场景识别（深夜 + 学习） |
| 2 | 点击第一首的 **喜欢** 👍 | Toast 显示：**已记录你的喜欢，后续会为你推荐更多类似的歌曲**<br>终端日志：`[FEEDBACK] like track_id=xxx` | 展示 like 文案正确<br>展示终端日志 |
| 3 | 点击第三首的 **不喜欢** 👎 | Toast 显示：**已记录你的不喜欢，后续推荐将排除这首歌**<br>该歌曲从列表消失（或标记为已记录） | 展示 dislike 排除逻辑 |
| 4 | 发送：**我想要风格的** | Loading 动效<br>返回爵士风格的安静音乐<br>查询中加入"爵士" | 展示曲风细化 |
| 5 | 点击 **换一批** 🔄 | Loading 动效<br>返回 5 首**完全不同**的歌曲（上一批的不会出现） | 展示换一批去重逻辑 |
| 6 | 发送：**不要太安静的，稍微有点节奏感** | Loading 动效<br>返回能量度中等的爵士乐<br>energy 从 low → medium | 展示能量偏好调整 |
| 7 | 点击第二首的 **喜欢** 👍 | Toast 显示正确文案<br>终端日志显示 like | 展示第二次 like |
| 8 | 发送：**再来 3 首类似的** | Loading 动效<br>返回 3 首（top_k=3）<br>风格延续上一轮偏好 | 展示数量控制 + 风格延续 |

**演示台词建议**：
> "大家可以看到，当我点击喜欢时，系统提示会推荐更多类似的歌曲，而不是排除。点不喜欢时才会排除。换一批后，之前推荐的歌曲不会再出现，这是因为系统自动排除了上一批结果。"

## 流程二：运动场景 - 高能量摇滚

**演示重点**：场景切换 → 能量偏好 → 声乐类型 → 多次换一批

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**推荐适合跑步听的歌，要高能量的** | Loading 动效<br>返回 5 首高能量/节奏感强的音乐<br>scene=运动，energy=high | 展示场景 + 能量识别 |
| 2 | 点击第一首的 **喜欢** 👍 | Toast + 终端日志 | 展示 like 记录 |
| 3 | 点击 **换一批** 🔄 | 返回 5 首不同的歌（无重复） | 展示去重 |
| 4 | 点击 **换一批** 🔄（再次） | 返回 5 首不同的歌（前 10 首都不出现） | 展示 exclude_ids 累积 |
| 5 | 发送：**不要摇滚，要电子音乐** | Loading 动效<br>返回电子音乐（EDM/Electronic）<br>genre 更新 | 展示曲风切换 |
| 6 | 点击第三首的 **不喜欢** 👎 | Toast + 该歌曲排除 | 展示 dislike |
| 7 | 发送：**纯音乐更好** | Loading 动效<br>返回无歌词的电子乐<br>vocals=instrumental | 展示声乐类型偏好 |
| 8 | 点击 **换一批** 🔄 | 返回新的纯音乐电子乐 | 展示偏好延续 |

**演示台词建议**：

> "注意看我连续点了两次换一批，这 15 首歌都完全不重复。系统会累积排除历史推荐的歌曲，确保每次都有新鲜感。"

## 流程三：情感场景 - 治愈系法语香颂

**演示重点**：模糊需求 → 多轮澄清 → 语言偏好 → 喜欢影响推荐

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**我今天心情不太好，推荐点治愈的歌** | Loading 动效<br>返回 5 首温暖治愈风格的音乐<br>mood=治愈 | 展示情绪识别 |
| 2 | 发送：**有没有法语的歌？** | Loading 动效<br>返回法语香颂/流行乐<br>query 中加入"法语" | 展示语言偏好 |
| 3 | 点击第一首的 **喜欢** 👍 | Toast + 终端日志 | 记录 like |
| 4 | 点击第二首的 **不喜欢** 👎 | Toast + 排除 | 记录 dislike |
| 5 | 发送：**类似第一首那种风格的** | Loading 动效<br>基于 liked_songs 提取风格<br>返回类似风格的法语歌 | 展示 liked 影响推荐 |
| 6 | 点击 **换一批** 🔄 | 返回新的法语治愈歌曲 | 展示去重 |
| 7 | 发送：**钢琴伴奏的更好** | Loading 动效<br>返回钢琴伴奏的法语歌<br>query 中加入"钢琴" | 展示乐器偏好 |
| 8 | 点击第一首的 **喜欢** 👍<br>再次点击 **换一批** 🔄 | Toast + 终端日志<br>返回新结果 | 展示完整反馈链路 |

**演示台词建议**：
> "当我点了一首喜欢的歌后，后续推荐会参考这首歌的风格。大家可以看到终端日志，每次反馈都有记录，而且 like 和 dislike 的处理逻辑是不同的。"

## 流程四：艺术家排除与多样性控制

**演示重点**：艺术家拒绝 → 自动排除 → 多样性展示 → 完整反馈闭环

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**推荐一些器乐爵士，适合深夜独处思考** | Loading 动效<br>返回 5 首器乐爵士<br>mood=思考, scene=深夜, vocals=instrumental | 展示复杂意图识别 |
| 2 | 观察推荐结果中的艺术家分布 | 可能返回多首同一艺术家的作品（如 Blue Dot Sessions） | 指出艺术家重复现象 |
| 3 | 发送：**不要 Blue Dot Sessions 了，换个艺术家** | Loading 动效<br>终端显示意图识别：排除艺术家<br>返回不同艺术家的器乐爵士 | **展示艺术家排除机制** |
| 4 | 点击第一首的 **喜欢** 👍 | Toast + 终端日志 | 记录 like |
| 5 | 点击 **换一批** 🔄 | 返回新的器乐爵士<br>**Blue Dot Sessions 不再出现** | 展示艺术家排除持续生效 |
| 6 | 发送：**再换一批，还是要不同的艺术家** | Loading 动效<br>返回另一批不同艺术家的作品<br>保持爵士 + 器乐偏好 | 展示多样性控制 |
| 7 | 点击第三首的 **不喜欢** 👎 | Toast + 排除该歌曲 | 展示歌曲级别排除 |
| 8 | 发送：**再来 3 首类似的** | 返回 3 首器乐爵士<br>排除已推荐/拒绝的歌曲和艺术家 | 展示完整排除闭环 |

**演示台词建议**：
> "大家注意看，当我明确说'不要 Blue Dot Sessions'后，系统把这个艺术家记录到排除列表。后续的换一批和新的推荐中，这个艺术家的歌都不会再出现。这是最近新增的艺术家级别排除功能，能避免同一个艺术家刷屏。"

## 流程五：跨语言检索能力

**演示重点**：中文查询 → 英文元数据 → BGE-M3 跨语言对齐

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**推荐适合学习的纯音乐** | Loading 动效<br>返回器乐作品（可能包含英文艺术家名） | 展示中文查询正常工作 |
| 2 | 发送：**我想要低保真风格的** | Loading 动效<br>返回 Lo-fi 风格音乐<br>推荐理由中可能出现"低保真采样"等描述 | 展示中文术语识别 |
| 3 | 观察推荐理由 | 理由包含具体音乐特征描述<br>如"弱化节奏推进，合成器铺陈出沉浸感" | **展示推荐理由质量提升** |
| 4 | 发送：**有点 ambient 氛围感的** | Loading 动效<br>返回 Ambient 风格音乐<br>中英文混合查询正常处理 | 展示中英混合查询 |
| 5 | 点击第一首的 **喜欢** 👍 | Toast + 终端日志 | 记录 like |
| 6 | 发送：**类似这种，但要更 jazz 一点** | Loading 动效<br>返回 Ambient Jazz 融合风格<br>推荐理由体现跨风格特征 | 展示风格融合检索 |
| 7 | 点击 **换一批** 🔄 | 返回新的 Ambient Jazz | 展示去重 + 风格延续 |
| 8 | 发送：**为什么推荐这首歌？** | 返回这首歌的推荐理由<br>包含具体的音乐特征描述 | **展示推荐理由的可解释性** |

**演示台词建议**：
> "系统底层用的是 BGE-M3 多语言模型，可以理解中文查询并匹配英文元数据。大家看推荐理由，不是简单的'适合学习'这种空话，而是会说'弱化节奏推进，合成器铺陈出沉浸感'这种有信息量的描述。这是最近优化的离线证据增强功能。"

## 流程六：音频试听功能

**演示重点**：试听可用性 → 播放控制 → 反馈闭环

| 轮次 | 用户操作 | 预期系统行为 | 演示要点 |
|------|---------|-------------|---------|
| 1 | 发送：**推荐一些适合学习的轻音乐** | Loading 动效<br>返回 5 首音乐<br>部分歌曲显示"可试听"标签 | 展示试听标签 |
| 2 | 观察推荐卡片 | 部分卡片有播放按钮<br>标签显示"可试听" | 指出试听功能入口 |
| 3 | 点击第一首的 **播放按钮** ▶️ | 底部出现音频播放器<br>开始播放<br>播放按钮变为暂停 | **展示音频播放功能** |

## 演示前准备检查清单

### 环境准备

```bash
# 1. 确保后端服务运行
python scripts/run_api.py

# 2. 确保前端服务运行
cd frontend && npm run dev

# 3. 终端开两个窗口
窗口 1: 后端日志（看 [FEEDBACK] 日志）
窗口 2: 前端浏览器控制台（看网络请求）
```

### 功能验证

**基础功能**
- [ ] Loading 动效正常（5 阶段 × 3s，最后阶段 dots 跳动）
- [ ] like toast 文案："已记录你的喜欢，后续会为你推荐更多类似的歌曲"
- [ ] dislike toast 文案："已记录你的不喜欢，后续推荐将排除这首歌"
- [ ] refresh toast 文案："已排除上一批结果，正在基于你当前偏好换一批推荐..."
- [ ] 换一批后无重复歌曲
- [ ] 终端日志显示 `[FEEDBACK] like/dislike/refresh track_id=xxx`

**新增功能**
- [ ] 艺术家排除：说"不要某艺术家"后，该艺术家的歌不再出现
- [ ] 推荐理由包含具体音乐特征（不是"适合学习"这种空话）
- [ ] 音频试听：部分歌曲显示"可试听"标签，点击可播放
- [ ] 播放器控制：播放/暂停、切换歌曲正常工作
- [ ] 跨语言查询：中文查询能匹配英文艺术家名

### 演示技巧

1. **Loading 阶段停顿**：发送消息后等 2-3 秒，让观众看到"正在分析用户场景..."→"正在分析用户情绪..."的切换
2. **终端日志同步**：演示 like/dislike 时，切换到终端窗口展示日志输出
3. **换一批对比**：换一批前让观众记住第一首歌，换完后确认那首歌不再出现
4. **liked 影响推荐**：点喜欢后，下一轮推荐时指出"这首歌的风格和刚才你喜欢的那首类似"

## 常见问题应对

| 问题 | 应对方案 |
|------|---------|
| Loading 动效不显示 | 检查前端控制台是否有报错，确认 `isSending` 状态正常 |
| 换一批出现重复歌曲 | 检查后端日志，确认 `last_recommendation` 不为空 |
| 终端没有反馈日志 | 确认后端服务以 INFO 级别运行，检查 `logger.info` 调用 |
| Toast 文案不正确 | 清除浏览器缓存，重新加载页面 |
| LLM 响应超时（>30s） | 切换到 mock 模式演示，或检查 DashScope API 连接 |
| 艺术家排除不生效 | 确认说的是"不要XX艺术家"句式，检查终端日志是否有艺术家排除意图 |
| 推荐理由太简单 | 切换到 Qwen 模式（非 mock），检查是否走了 fallback |
| 音频无法播放 | 检查后端 `/audio/` 路由是否正常，检查文件是否存在 |
| 同一艺术家反复出现 | 使用流程四演示艺术家排除功能 |
