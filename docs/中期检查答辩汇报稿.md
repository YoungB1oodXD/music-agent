## 一、项目背景与目标

随着音乐流媒体平台的普及，用户面临着海量曲库带来的"信息过载"问题。传统的推荐列表虽然有效，但缺乏深度交互和意图理解，难以捕捉用户复杂、动态且带有情感色彩的需求。

本课题的目标是构建一个能够通过自然语言多轮对话，深入理解用户情境与偏好，并提供精准、可解释音乐推荐的智能助手。

核心功能包括：
1. 多轮对话状态管理
2. 语义检索与协同过滤的混合推荐
3. 基于大语言模型的意图理解与推荐解释生成
4. 用户反馈闭环

## 三、已完成工作详情

### 3.1 数据层（100% 完成）

已完成 FMA 音乐元数据与 Last.fm 用户行为数据的清洗与融合工作：

- 处理歌曲数量：106,573 首
- Tags 覆盖率：从 0% 修复至 97.5%
- 向量索引规模：106,573 条，约 2GB
- 构建了 8,000 首可播放歌曲的音频映射

关键技术改进：移除了 Last.fm 外部 API 依赖，改为 FMA 单源架构，解决了校园网环境下的运行稳定性问题。

### 3.2 语义检索引擎（100% 完成）

采用 BGE-M3 多语言向量模型与 ChromaDB 向量数据库，实现了：

- 支持中英文自然语言查询
- 支持情感化、场景化描述检索
- 检索响应时间小于 100 毫秒

技术特点：通过离线加载模型，不依赖外网，可完全在本地环境运行。

### 3.3 协同过滤推荐（100% 完成）

基于 Implicit ALS 算法，利用 839,122 条 Last.fm 用户行为数据进行训练：

- 模型参数：64 维隐向量，15 次迭代
- 支持基于歌曲的相似推荐
- 已解决 Windows 环境下的 BLAS 兼容性问题

### 3.4 大语言模型集成（100% 完成）

接入阿里云 DashScope 的 Qwen3.5-plus 模型，实现了：

- 基于 JSON Schema 的结构化意图识别
- 多轮对话状态追踪
- 结合 RAG 上下文的推荐理由生成

同时提供了 Mock 模式，确保在无 API Key 环境下核心逻辑可验证。

### 3.5 Agent 编排系统（100% 完成）

实现了完整的 Agent 编排流程：

用户输入 → 意图识别 → 槽位提取 → 工具调度 → 结果生成 → LLM 解释

支持多轮对话状态管理、偏好累积、反馈闭环等核心功能。

### 3.6 前后端系统（100% 完成）

**后端**：基于 FastAPI，实现了 /chat、/feedback、/audio 等接口，支持会话状态管理。

**前端**：基于 React + Vite + Tailwind CSS，实现了对话界面、推荐卡片、反馈按钮、Toast 提示等完整交互功能。

## 四、核心创新点

### 4.1 双脑架构设计

将推荐系统分为语义理解（左脑）和行为学习（右脑）：

- 左脑：BGE-M3 + ChromaDB，理解自然语言查询意图
- 右脑：Implicit ALS，基于用户行为的相似推荐

两路召回相互补充，提升推荐的精准度与个性化程度。

### 4.2 Agent + RAG 架构

采用 LLM 驱动的 Agent 编排：

- 意图识别 → 工具调度 → RAG 增强 → LLM 生成
- 多轮对话状态追踪
- 反馈闭环影响后续推荐

### 4.3 离线模型本地化

- BGE-M3 模型离线加载，不依赖外网
- ChromaDB 本地持久化
- 完全适配校园网环境运行

## 五、当前存在的问题

### 5.1 音频试听功能未完全实现

**问题**：FMA_small 音频文件与推荐结果 ID 对齐存在部分问题。

**影响**：部分歌曲无法正常播放。

**处理**：已作为系统局限性说明，答辩时将诚实汇报。

### 5.2 Hybrid 推荐融合率受限

**问题**：FMA 与 CF 模型数据源独立，ID 空间不重叠。

**影响**：混合推荐退化为语义搜索为主。

**处理**：已在报告中说明，作为后续优化方向。

### 5.3 缺少离线评估指标

**问题**：尚未完成 NDCG、Precision@K 等离线评估。

**影响**：无法量化推荐质量。

**计划**：答辩时说明为后续优化方向。
